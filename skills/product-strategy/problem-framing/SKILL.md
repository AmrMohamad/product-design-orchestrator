# Problem Framing

## Mission

Define the problem so multiple solutions remain possible and the team can decide whether to proceed.

## Use this skill when

- frame problem
- design brief
- feature request
- scope product work

## Inputs

- stakeholder request
- discovery evidence
- constraints
- business/service context

## Protocol

1. Write the current situation, affected users/actors, obstacle, consequence, and evidence.
2. Separate root problem, symptoms, and requested solution.
3. State desired user and product/service outcomes plus harm/quality guardrails.
4. Set in-scope journey and explicit non-goals.
5. List constraints and classify as legal/safety, technical, operational, policy, time, assumption, or preference.
6. Map assumptions by uncertainty × consequence and identify what could invalidate the work.
7. Create 3–5 experience principles specific enough to reject a design.
8. Define proceed/reframe/stop and review triggers.

## Required outputs

- problem statement
- outcomes/guardrails
- scope/non-goals
- assumption map
- experience principles

## Quality bar

- The statement does not name the proposed UI as the need.
- Evidence and assumptions are separate.
- Scope covers a complete user outcome.
- Principles create real trade-offs.

## Failure modes to avoid

- "Users need a dashboard" as a problem.
- Scope by screen count.
- Generic principles like easy/modern.
- Ignoring operational consequences.

## Handoff

Record consequential decisions and pass only decision-relevant context to the next phase.
