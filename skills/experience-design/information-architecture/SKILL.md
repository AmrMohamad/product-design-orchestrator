# Information Architecture

## Mission

Help people know where they are, what exists, and how to find or understand it as content grows.

## Use this skill when

- information architecture
- navigation
- taxonomy
- sitemap
- content model

## Inputs

- content/domain inventory
- user tasks/mental models
- existing analytics/search logs
- platform constraints

## Protocol

1. Inventory content, objects, actions, attributes, relationships, ownership, lifecycle, and permissions.
2. Model user concepts separately from database or organizational structure.
3. Define taxonomy and naming principles; resolve ambiguity, synonyms, and overloaded terms.
4. Choose navigation model and levels based on task frequency, scope, and orientation needs.
5. Design browse, search, filtering, cross-linking, and recovery routes.
6. Map page/screen hierarchy and content priority for key tasks.
7. Test high-risk labels/structure with card sort, tree test, first-click, search logs, or usability evidence.
8. Plan URL/deep-link, content growth, archiving, localization, and component reuse.

## Required outputs

- domain/content model
- taxonomy/labels
- navigation/sitemap
- page hierarchy
- IA validation plan

## Quality bar

- Labels use user language.
- Navigation supports orientation and frequent tasks.
- Structure survives content growth.
- Database/org chart does not dictate UX without reason.

## Failure modes to avoid

- Mega menus before content modeling.
- Every object as a top-level tab.
- Vague labels like Solutions/Resources.
- Testing IA only through stakeholder preference.

## Handoff

Record consequential decisions and pass only decision-relevant context to the next phase.
