# Content Design

## Mission

Use content as part of the product behavior, not filler added after layout.

## Use this skill when

- UX writing
- microcopy
- content design
- error messages
- labels

## Inputs

- user job/context
- domain/legal language
- flow/states
- brand voice
- localization/accessibility needs

## Protocol

1. Identify the decision or action each content element must support.
2. Use the user's vocabulary while preserving necessary domain precision.
3. Create a content hierarchy: promise/outcome, key facts, action, supporting detail, help.
4. Write specific labels and CTAs that describe the result, not generic "Continue" when avoidable.
5. Explain errors with what happened, affected data, recovery, and retained progress.
6. Write confirmations and destructive prompts around consequence and object, not fear.
7. Design empty, loading, permission, notification, onboarding, and success content as real states.
8. Test comprehension, scanning, translation expansion, numeracy, dates/currency, and screen-reader wording.
9. Create reusable terms and message patterns; involve legal/compliance without publishing internal language unchanged.

## Required outputs

- content hierarchy
- key copy and state messages
- terminology model
- content acceptance criteria
- localization notes

## Quality bar

- Every string supports understanding/action.
- Tone matches risk and context.
- Errors are actionable.
- Real content fits the design.

## Failure modes to avoid

- Lorem ipsum for final hierarchy.
- Cleverness in high-stress moments.
- Generic CTAs.
- Blaming users.
- Hiding material terms.

## Handoff

Record consequential decisions and pass only decision-relevant context to the next phase.
