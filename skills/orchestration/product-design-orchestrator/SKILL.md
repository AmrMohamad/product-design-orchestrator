# Product Design Orchestrator

## Mission

Choose and sequence the lightest credible lifecycle, specialist hats, artifacts, and quality gates for the product decision.

## Use this skill when

- full product design
- new feature
- greenfield
- redesign
- multi-phase design and build

## Inputs

- user request and success condition
- repository and product context
- risk/domain constraints
- existing .design artifacts

## Protocol

1. Classify the task as patch, component, feature, redesign, greenfield, reference rebuild, research, system, or live optimization.
2. Inspect available context before asking for information already present in code, analytics, docs, screenshots, or prior decisions.
3. Write the Design Read and identify the decision that must be made now.
4. Map uncertainty and risk; select lifecycle phases that cannot responsibly be compressed.
5. Route each phase to the narrowest skill owner; keep simultaneous craft context to three or fewer skills.
6. Define phase gates and only create artifacts that enable a decision or preserve continuity.
7. Maintain an evidence ledger, assumption map, and decision record for consequential work.
8. Deliver vertical slices and return to earlier phases when rendered behavior invalidates an assumption.
9. Close with verified outcomes, unresolved uncertainty, and the next evidence trigger.

## Required outputs

- design read
- selected mode and phase map
- skill plan
- artifact/gate plan
- completion report

## Quality bar

- Process is proportional to risk and reversibility.
- Every phase names a decision and exit gate.
- Research, design, and implementation remain traceable.
- The orchestrator does not replace specialist reasoning with generic checklists.

## Failure modes to avoid

- Running every phase for a trivial patch.
- Starting visual design before understanding the product/system.
- Loading every available skill.
- Treating a polished artifact as evidence.

## Handoff

Record consequential decisions and pass only decision-relevant context to the next phase.
