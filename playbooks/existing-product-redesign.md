# Playbook — Existing-product redesign

A redesign starts with diagnosis, not a new aesthetic.

1. Run `project-inspector`; trace real journeys, code, data, roles, analytics, support signals, tokens/components, tests, and rendered states.
2. Separate broken behavior, system drift, content debt, accessibility/performance defects, product-model problems, and subjective visual opportunity.
3. Record what works and must be preserved. Decide preserve/evolve/replace for each affected layer.
4. Frame the outcome and baseline; avoid “make it modern” as the goal.
5. Research the actual pain/risk where existing evidence is insufficient.
6. Produce 2–3 directions only when the trade-off is consequential. Translate references into principles rather than costumes.
7. Choose a migration path: in-place correction, progressive shell/system change, route-by-route replacement, or deliberate rebuild.
8. Implement vertical slices behind flags when risk warrants; keep URLs, data, permissions, semantics, analytics, and critical workflows stable.
9. Compare old/new tasks, not just screenshots. Run visual, functional, accessibility, performance, content, localization, and regression checks.
10. Release with metrics, support plan, rollback, and follow-up research. Remove old system debt deliberately.
