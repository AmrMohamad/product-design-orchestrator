# Playbook — Design system creation or evolution

1. Inventory real product surfaces, primitives, tokens, component variants/states, duplicate patterns, code usage, accessibility, and ownership.
2. Define product principles and adoption outcomes before drawing a component library.
3. Decide the foundation: extend current system, adopt an official platform/system, or build a thin owned layer. Avoid mixed primitive systems.
4. Establish primitive, semantic, and component tokens with modes and migration strategy.
5. Prioritize patterns/components by product frequency, risk, inconsistency, and leverage—not showcase appeal.
6. Specify behavior, content, responsive/platform, accessibility, theming, localization, and test contracts.
7. Implement docs/examples and visual/accessibility/component tests beside code.
8. Pilot on real journeys; measure adoption, defects, contribution time, drift, and user-quality outcomes.
9. Govern proposals, review, versioning, deprecation, migration, and ownership.
10. Treat the system as product infrastructure: maintain from field evidence and retire unused complexity.
