# Documentation Governance

## Mission

Keep the system useful and trusted after its first release instead of becoming a static gallery or bottleneck.

## Use this skill when

- design system docs
- governance
- component documentation
- design ops

## Inputs

- system inventory
- team structure
- release process
- consumer needs
- usage/adoption data

## Protocol

1. Define audiences and the decisions documentation must support.
2. Create consistent docs for foundations, components, patterns, accessibility, content, code, and migration.
3. Name owners, maintainers, contribution path, review criteria, RFC/decision process, and support channel.
4. Version changes by behavioral/visual impact; publish changelog and migration guidance.
5. Deprecate with replacement, rationale, telemetry/usage, timeline, and compatibility plan.
6. Measure adoption, duplication, exceptions, issue resolution, accessibility defects, and consumer satisfaction.
7. Run regular system audits and prioritize debt by product consequence.
8. Allow justified escape hatches and feed repeated exceptions back into the system.

## Required outputs

- documentation model/templates
- ownership/contribution model
- release/deprecation policy
- adoption metrics
- audit cadence

## Quality bar

- Docs explain when/why, not just props.
- Governance enables contribution.
- Breaking changes are explicit.
- Adoption is measured by outcomes, not coercion.

## Failure modes to avoid

- Central team as gatekeeper.
- Docs copied from code comments only.
- Silent visual changes.
- Never deleting obsolete components.

## Handoff

Record consequential decisions and pass only decision-relevant context to the next phase.
